## Шаг 0.5 — Методологическая проверка (ПЕРЕД языковой редактурой)

**Только для project=factor. Выполнять до любой языковой правки.**

Прочитать:
- `factor/data/review-patterns.md` — типичные ошибки из ревью (разделы 11-13 — самые свежие)
- `factor/data/factor-methodology.md` — правильная методология: воронка, показатели, KPI, аутрич

**Если тема статьи касается показателей / KPI / воронки / плана продаж / аутрича / управления ОП — выполнить чеклист:**

- [ ] Порядок воронки: **лид → квал → КЭВ → продажа** (не лид → КЭВ → квал → оплата)
- [ ] KPI = фокусные показатели под цели, к которым привязывают мотивацию (не «все метрики»)
- [ ] Дневные метрики включают **счета и оплаты** (не только активности — это первая половина воронки)
- [ ] Еженедельно: цикл сделки **не анализируется** (только ежемесячно)
- [ ] NPS — **не показатель ОП**: убрать из любого перечня метрик ОП
- [ ] Холодный аутрич: первый шаг — **звонок с выходом на ЛПР**, не email/TG
- [ ] «Буфер 10–15%» от суммы личных планов — **не рекомендовать**
- [ ] «Счетов выставлено» — **не** «КП/КП отправлено»
- [ ] «Грязная деталь» — **не должно быть в тексте** статьи (внутренний редакторский термин)

**Формат при нахождении методологических ошибок:**
- В Google Doc: добавить комментарий с пометкой `🚨 Методология: [описание ошибки]`
- Если правка однозначная (термин → термин) — исправить через replace-text
- Если требует переписки раздела — пометить `⚠️ [нужна переработка раздела по методологии]`, не переписывать самостоятельно

**Приоритет:** методологические ошибки исправить первыми, языковые — после.

**Текущий год:** бери из `memory/MEMORY.md` раздел `# currentDate`. Если в статье год не совпадает с текущим — исправить в H1, SEO Title, cover_tag.

## Правки по приоритетам

### 1. Убить ИИ-обороты — форма vs функция

**Слоп = форма без функции.** Перед удалением AI-паттерна проверить, что за ним стоит:

1. Найти AI-паттерн («следует отметить», «важно понимать», «таким образом»...)
2. Посмотреть на следствие: есть ли за фразой число / кейс / причинно-следственная связь?
3. **Есть содержание** → убрать только фрейм, содержание оставить:
   «Следует отметить: 7 из 10 РОПов не смотрят CRM ежедневно» → «7 из 10 РОПов не смотрят CRM ежедневно»
4. **Нет содержания** → убрать вместе со следствием, абзац целиком:
   «Следует отметить, что данный подход является эффективным» → удалить

Механическое удаление по списку даёт ложные срабатывания: фраза с сильным содержанием теряет смысл, а пустая фраза без фрейма остаётся пустой. Сначала функция, потом форма.

Найди каждое вхождение и примени правило выше:

| Было | Стало |
|------|-------|
| следует отметить | убрать |
| важно понимать | убрать |
| давайте разберёмся | убрать |
| таким образом | убрать или «итого:» |
| комплексный подход | описать конкретно, что делается |
| эффективный инструмент | назвать инструмент + цифра результата |
| вывести на новый уровень | конкретный результат («+40% конверсии») |
| в современном мире | убрать |
| не секрет, что | убрать |
| очевидно, что | убрать |
| данный | «этот» или переформулировать |
| является | заменить глаголом действия |
| осуществляется | заменить активным глаголом |
| очень, действительно, крайне | убрать |
| Здесь: [что будет в тексте] | «В этой статье:» или убрать |
| [предмет] смотрит / видит / считает (неодушевлённый) | заменить: «аудит проверяет», «система показывает» |
| битва / выиграть / проиграть (военные метафоры в B2B) | убрать, заменить на нейтральное |
| выше воронки / ниже воронки | «за пределами воронки продаж» / «на поздних этапах» |
| тонет / исчезает / растворяется (о письме/КП) | «остаётся без ответа» |

### 1б. Кальки с английского — думать, не сверять по списку

**Что такое калька:** фраза, которая переведена буквально с английского и не существует в живом русском деловом языке. AI генерирует их постоянно, потому что обучен на двуязычных текстах.

**Диагностический вопрос для каждой фразы:**
> Сказал бы так РОП в WhatsApp коллеге? Если нет — скорее всего, калька.

Второй тест:
> Если перевести эту фразу на английский — она звучит естественно? Тогда это, возможно, обратная калька.

**Как думать, а не сверять:**
Прочитай каждое предложение и спроси себя: это написано по-русски или по-английски с русскими словами? Особое внимание — к глаголам с абстрактными существительными и к физическим глаголам у нефизических понятий.

**Конкретные примеры из реальных статей — учить паттерн, не конкретную фразу:**

| Калька | Почему плохо | Правка |
|--------|-------------|--------|
| «продажи стоят там же» | sales are stagnant → физический глагол у абстрактного понятия | «план по-прежнему не выполняется» |
| «разовый снимок» | snapshot → прямой перевод, не существует в русском | «разовый аудит», «разовая диагностика» |
| «что делает результат» | what makes the result → калька вопроса | «как сделать результат», «что даёт результат» |
| «числа» вместо «показатели» | numbers → в русских продажах говорят «показатели», «цифры» в конкретном контексте, не «числа» | «выходит на план», «даёт те же показатели» |
| «двигать сделку» | move a deal → | «вести сделку», «продвигать клиента» |
| «закрыть боль» | close the pain → | «решить проблему», «снять возражение» |
| «делать числа» | make numbers → | «выполнять план», «делать показатели» |
| «стек инструментов» | tool stack → | «набор инструментов», «инструменты» |
| «реагировать на [период/результат]» | react to → по-русски не реагируют на следующий месяц | «учитывать в следующем месяце», «корректировать по итогам» |
| «отреагировать на ситуацию» | respond to the situation → | «скорректировать курс», «разобраться с ситуацией» |

**Частый класс — физические глаголы у нефизических понятий:**
«продажи стоят / висят / лежат / падают» как физические объекты — не по-русски. В русском деловом языке абстрактные понятия *изменяются, растут, снижаются, выполняются* — не «стоят» и не «лежат».

**T3-источники + конкретная цифра = запрещено:**
«По оценкам рынка, менее 10%» — если нет верифицированного источника (T1/T2), конкретную цифру ставить нельзя. T3 даёт только размытые формулировки: «большинство», «единицы», «редко». Цифра без источника — выдумка.

### 2. Иностранные термины и бренды → русский язык

Аудитория — обычный продавец, не технический специалист. Английские термины, названия брендов и аббревиатуры заменять на понятные русские описания.

**Правило:** если обычный продавец WB не знает что это — убрать или расшифровать.

**Правило про цитаты:** цитата — не повод оставлять английский. Переводить смысл, а не слова буквально. Источник и авторство сохранять, язык — русский.

**Примеры замен:**

| Было | Стало |
|------|-------|
| UiPath Autopilot, Automation Anywhere AI Agent Studio | «крупные платформы автоматизации» |
| LLM-рассуждения | «понимание смысла текста», «логика языковой модели» |
| Rate limits API | «лимиты запросов к API» |
| Predictions 2025, Forecasts | «прогноз на 2025 год» |
| reasoning (в цитате) | «осмысленных рассуждений» |
| Application layer (в цитате) | «прикладной уровень» |
| ROI | «окупаемость», «возврат инвестиций» |
| prompt → промпт | **оставить** — слово понятно широкой аудитории |
| Pipeline | «процесс», «цепочка задач» |
| Guardrails | «ограничения», «защита от ошибок» |
| Token (API) | «ключ доступа», «API-ключ» |
| Framework | «платформа», «инструмент» |
| churn rate | «процент оттока», «доля потерянных клиентов» |
| SQL (Sales Qualified Lead) | «квалифицированный лид» |
| Key Results (OKR) | «ключевые результаты» |
| Accountability Chart | «система подотчётностей» (название системы — оставить, термин — перевести) |

**Когда можно оставить термин:**
- Он объяснён в той же фразе: «CRM (система учёта клиентов)»
- Без него теряется точность, и аудитория его знает (API, SEO, WB, DRR, промпт)
- В заголовке ключевого запроса (SEO-необходимость)

**Не сохранять:** названия конкурирующих продуктов и платформ без объяснения контекста — это рекламирует их, не помогает читателю.

### 3. Расплывчатость → числа

«многие», «часто», «некоторые», «как правило», «в большинстве случаев» — либо конкретная цифра, либо «в 7 из 10 проектов», либо пример из кейса.

### 3. Пассив → актив

- «была проведена работа» → «мы разобрали»
- «было выявлено» → «нашли»
- «позволяет улучшить» → конкретный результат

### 4. Вводный абзац — TL;DR формат

Первый абзац выполняет две функции: удерживает читателя и цитируется Google AI Overview. Это одно требование, если написать правильно.

**Структура TL;DR-лида:**
```
[ключ + прямой ответ] → [цифра из опыта] → [контр-интуитивная позиция]
```

**Почему это критично:** 44% AI-цитат берётся из первых 30% текста. Google AI извлекает блоки 130–160 слов, которые должны быть самодостаточными — содержать ответ + факт + контекст без опоры на остальной текст.

**Проверка редактора:** Можно ли взять первый абзац и вставить его в AI-ответ без контекста остальной статьи? Если нет — переписать.

❌ «Конверсия в продажу — это отношение числа сделок к числу лидов...» (определение)
❌ «Сегодня разберём, как построить отдел продаж с нуля...» (анонс)
❌ «Клиент прислал мне результаты аудита...» (вымышленная личная история)
❌ «Многие предприниматели сталкиваются с тем, что...» (вводная вода)

✅ «Отдел продаж с нуля строится минимум за 3 месяца — и это если процесс выстроен правильно. В 200+ проектах типичный провал выглядел одинаково: наняли людей, дали CRM, ждали результат через месяц. Причина всегда одна — без скриптов, воронки и разбора звонков даже сильный менеджер работает вполсилы.»
✅ «Конверсия 4% — клиент считал это нормой. Мы открыли CRM и нашли причину за 20 минут: менеджеры не перезванивали лидам дольше суток.»

Если первый абзац — вода или анонс, переписать по формату: **[ключ + ответ] → [цифра] → [контр-интуитивная позиция]**. Голос — компания («мы»), не личная история.

### 5. Голос компании, не личный нарратив

Статья пишется от лица компании Фактор Продаж («мы»), не от первого лица Кандеева как частного человека.

**Запрещено:**
- Вымышленные личные истории: «Клиент прислал мне...», «Я увидел в его CRM...»
- Личный нарратив с «я»/«мне» — даже если звучит живо, это художественный вымысел

**Разрешено:**
- «В наших проектах мы видим...»
- «Когда клиент показывает нам воронку...»
- «Мы против [X], потому что [Y]»

Кандеев — автор по имени. Но голос — голос компании с 200+ проектами, а не личный блог.

2–3 места, где слышна позиция ФП — не «эксперты считают», а «мы видим», «у нас в проектах».

Шаблоны:
- «У нас в практике — [факт из кейса]»
- «Когда клиент говорит [цитата] — как правило, это [диагноз]»
- «Мы против [X], потому что [Y]. Вместо этого — [Z].»

### 5а. Источники — расшифровывать при первом упоминании

Если в тексте назван внешний источник (Gartner, McKinsey, HubSpot, CSO Insights и т.д.) — при первом упоминании добавить короткую расшифровку в скобках.

| Было | Стало |
|------|-------|
| По данным Gartner, ... | По данным Gartner (международная исследовательская компания), ... |
| McKinsey выяснили, ... | McKinsey (глобальная консалтинговая компания), ... |
| По исследованию HubSpot, ... | По исследованию HubSpot (крупнейший B2B SaaS-провайдер), ... |

Правило: читатель блога — собственник или РОП, не аналитик. Он не обязан знать аббревиатуры и названия западных компаний.

### 6. Кейс — грязная деталь

Если в кейсе только цифры без истории — это шаблон, не кейс. Нужна одна конкретная деталь: что именно увидели, что сказал клиент, что было сломано.

**Не выдумывать.** Брать только из реальных кейсов на yoursite.ru/case/ или из company-context.md. Если деталей нет — оставить цифры как есть и пометить `[нужна деталь от клиента]`.

### 7. Ритм + burstiness-правка

После длинного предложения — одно из 4–6 слов. Намеренно, 3–4 раза в тексте.

«Это не работает.», «Проверяли. Не работает.», «Вот почему.»

Но осторожно: короткое предложение-резюме в конце абзаца легко становится AI-клише. Проверять: оно вытекает из предыдущего текста — или просто «поставлено для эффекта»?

❌ «Каждое предложение просто падало в ящик. Это другое.» — второе предложение не добавляет смысла
✅ «Каждое предложение падало в ящик и оставалось без ответа.» — один ёмкий образ вместо двух

**Измеримая проверка ритма** (тот же скрипт, что в /quality-gate М20–М22):

```bash
python3 scripts/text_quality_check.py /tmp/draft.md
```

- σ длины предложений ≥ 6 — норма; < 4,5 — текст монотонный, AI-ровный
- Скрипт выводит **monotone spans** — участки из 5+ предложений одной длины. Это адреса для правки: в каждом спане либо вставить ультракороткое предложение (3–5 слов), либо развернуть одно в 25+ слов
- Проверка после вставки: короткое предложение добавляет новую информацию? Если нет — это ритмический костыль, не настоящий ритм (см. выше)
- Human-контент даёт 5,44× больше органики (BrightEdge, 10M URL); задача редактора — не «убрать AI», а добавить редакторский ритм поверх черновика

### 7а. Стилистика русского языка

**Драматические слова в конце предложения — AI-маркер.**

Слова «провал», «дыра», «тонет», «рушится», «катастрофа» в концовке предложения создают искусственный накал. В живом тексте такие слова употребляются по делу, не для эффекта.

| Было | Стало |
|------|-------|
| «Но не с этого начинается провал.» | «Но корень проблемы — в другом.» |
| «Система рушится.» | «Конверсия падает, менеджеры теряют сделки.» |
| «Это дыра в воронке.» | «Здесь теряется каждый третий лид.» |

Правило: называть конкретное следствие, не общее слово-оценку.

**AI-фреймы — фразы, которые объясняют то, что только что было сказано.**

Эти фразы появляются, когда текст не доверяет читателю и перестраховывается:

❌ «Это не история про X. Это про Y.»
❌ «Речь не о том, чтобы X. Речь о Y.»
❌ «И дело не в том, что... Дело в том, что...»

Вместо объяснения — просто написать Y сразу, без предисловия.

**Глагол должен нести действие, а не заполнять место.**

- «аудит анализирует менеджеров» → «аудит проверяет *работу* менеджеров: скрипты, конверсию, дисциплину в CRM» (конкретика после двоеточия)
- «компания работает над проблемой» → «компания меняет скрипты / перестраивает воронку»
- «мы занимаемся построением ОП» → «мы строим отдел продаж»

**Называть предмет полностью, не обрывать фразу.**

Неполная мысль с подразумеваемым смыслом — признак AI-текста:
❌ «от разового аудита к системному контролю» → контролю *чего*?
❌ «мы работаем над качеством» → *чего именно*?
✅ «от разового аудита к системному контролю показателей продаж»

### 7б. Риторические приёмы — когда они не работают в B2B-тексте

AI-текст злоупотребляет ораторскими приёмами, которые органичны в речи или художественной прозе, но в деловой статье звучат надуманно. Тест: написал бы опытный РОП это в рабочем письме или в WhatsApp коллеге? Если нет — скорее всего, это ораторский приём не на своём месте.

**Антитеза «X, не Y» (два существительных через запятую)**

❌ «Это фотография, не система.»
❌ «Не отчёт — результат.»
❌ «Диагноз, а не лечение.»

Проблема: выглядит как афоризм, не как деловой текст. Противопоставление работает только если оба слова уже знакомы читателю и несут чёткий смысл. Иначе — умничание.

Как исправить: развернуть в предложение с конкретикой.
✅ «Аудит нашёл проблемы, но без плана внедрения они остались на бумаге.»

**Эллипсис — пропуск глагола для "удара"**

❌ «Не работает.» в конце абзаца без предшествующего контекста
❌ «Корень — в другом.» (в чём именно?)
❌ «Это другое.» (что именно другое?)

Эллипсис уместен, когда пропущенное слово очевидно из контекста. Когда читатель не понимает, что именно пропущено — это не удар, а обрыв.

Как исправить: либо написать полностью, либо убедиться, что пропуск не создаёт неоднозначности.

**Анафора — повтор слова в начале нескольких предложений подряд**

❌ «Не потому что менеджеры плохие. Не потому что продукт слабый. А потому что система не работает.»

Уместно в выступлении, в тексте — ощущение искусственно нагнетаемого ритма. Одного повтора достаточно. Три — уже приём ради приёма.

**Параллельные списки сильных глаголов**

❌ «Нашли. Разобрали. Исправили. Закрепили.»

Работает только если глаголы действительно описывают последовательные действия. Если это просто "перечень для ритма" — убрать, написать одним предложением.

**Общее правило:**
Риторический приём оправдан, если он помогает читателю понять быстрее — а не создаёт ощущение красивости. Проверка: убери приём, потеряет ли текст смысл? Если нет — приём лишний.

### 8. Тире — только там, где уместно в русском языке

**Уместно:**
- Именное сказуемое без глагола-связки: «Аудит — диагностика продаж»
- Список с опущенным глаголом: «Блок 1 — конверсия и выручка»
- Смысловой разрыв/неожиданность: «Позвонили — не взяли трубку»

**Не уместно (AI-паттерны с тире):**
- Вместо запятой перед «и»: «Нашли — и сразу исправили» → «Нашли и исправили»
- Переход между мыслями: «Менеджеры старались — результата не было» → «...старались, но результата не было»
- Антитеза «X — не Y»: «Начало — не конец», «Фотография — не система» → переписать развёрнуто
- Вместо двоеточия: «Вывод — нужно менять скрипты» → «Вывод: нужно менять скрипты»
- Для «удара» в конце: «Корень проблемы — в другом» → «Корень проблемы в другом»

**🔴 HARD GATE — выполнить обязательно после всех правок:**

Тире вредно везде — в прозе, в заголовках, в таблицах. Считать во всём тексте.

```python
# Для текста в памяти:
total = text.count('—')
print(f"Тире всего: {total}")  # должно быть ≤ 8

# Проверка по абзацам (≤ 2 в каждом):
for i, para in enumerate(text.split('\n\n')):
    c = para.count('—')
    if c > 2:
        print(f"Абзац {i}: {c} тире — НАРУШЕНИЕ: {para[:80]}")
```

Лимит: **не более 8 в статье суммарно (проза + таблицы + заголовки), не более 2 в абзаце.**

Замены:
- Перед пояснением/выводом → двоеточие: `Вывод — нужно менять` → `Вывод: нужно менять`
- Перед «но/однако» → запятая: `Старались — результата нет` → `Старались, но результата нет`
- В таблицах `<td>` → двоеточие: `Проблема — нет конверсии` → `Проблема: нет конверсии`
- Можно убрать совсем: `Корень — в другом` → `Корень в другом`
- **Оставить только**: именное сказуемое без связки: `KPI — инструмент` ✓

Вывести результат: `Тире после правки: X (лимит ≤ 8) ✓/✗`

### 9. Структурные AI-паттерны

Кроме запрещённых слов — есть запрещённые конструкции. Они не вычисляются поиском, но узнаются сразу.

**Эхо-объяснение** — сказать факт, потом перефразировать его своими словами:
❌ «57% пути клиент проходит сам. Это значит: к моменту звонка половина пути уже позади.»
Второе предложение не добавляет информации. Оставить только одно — то, которое точнее.

**Согласиться-оппонировать** — «X важно. Но/Однако...»:
❌ «Скрипты важны. Но дело не только в скриптах.»
AI обозначает «поворот» вместо того чтобы сразу писать суть. Убрать первую часть, начать со второй.

**Искусственная интрига** — «на первый взгляд», «казалось бы»:
❌ «Менеджеры, на первый взгляд, работали нормально.»
Сигнал читателю что сейчас будет разворот. Реальный текст описывает ситуацию, не анонсирует разворот.

**Сигнал-анонс вместо содержания** — «Вот что мы делаем», «Три ключевых момента»:
❌ «Вот что мы анализируем в каждом проекте:»
AI объявляет о том, что собирается сказать, перед тем как сказать. Убрать анонс, сразу перейти к содержанию.

**Параллельные риторические конструкции** — «Три ситуации — три решения», «Нашли. Разобрали. Исправили.»:
Хороши в речи. В деловом тексте — приём ради приёма.

**Эллипсис-удар в конце абзаца** — короткое предложение, которое резюмирует уже сказанное:
❌ «Это другой режим работы.» ❌ «Вот почему это важно.» ❌ «Работает.»
Оправдан только если добавляет новую информацию, а не подводит итог предыдущего.

## Шаг обязательный: Языковая чистка (раздел 9 референса)

**Это процедура, не декларация.** Перед финальным выводом пройди текст по каждому пункту — найди вхождения и исправь.

### Лексическая сочетаемость — найти и исправить
| Ошибка | Правка |
|--------|--------|
| «играть значение» | «иметь значение» или «играть роль» |
| «одержать успех» | «добиться успеха» или «одержать победу» |
| «предпринять меры» | «принять меры» или «предпринять шаги» |
| «большая половина» | «бо́льшая часть» |
| «оказать заботу» | «проявить заботу» |
| «поднять тост» | «произнести тост» |

### Плеоназмы — найти и удалить лишнее слово
Убирать второе слово, оно ничего не добавляет:
- «свободная вакансия» → «вакансия»
- «ведущий лидер» → «лидер»
- «главная суть» → «суть»
- «IT-технологии» → «технологии»
- «возвращаться обратно» → «возвращаться»
- «кивнул головой» → «кивнул»
- «первая премьера» → «премьера»
- «моя автобиография» → «автобиография»

### Управление — проверить предлоги
- «согласно / благодаря / вопреки» → дательный падеж (согласно **приказу**, НЕ «приказа»)
- «оплатить» → без «за» (оплатить **проезд**, НЕ «за проезд»)
- «по окончании / по приезде» → предложный (НЕ «по окончанию / по приезду»)
- «отзыв **о** книге» vs «рецензия **на** книгу» — не смешивать

### Деепричастные обороты — проверить субъект
Субъект деепричастия = субъект главного глагола. Если не так — переписать.
- ❌ «Проводя аудит, у нас возникли вопросы» → ✅ «Проводя аудит, мы столкнулись с вопросами»
- ❌ «Открыв воронку, стало ясно» → ✅ «Открыв воронку, мы увидели»

### Нанизывание родительных падежей
Цепочка из 3+ родительных — разбить или переформулировать:
- ❌ «вопрос повышения качества работы менеджеров отдела» → ✅ «как улучшить работу менеджеров»
- ❌ «система контроля выполнения плана продаж» → ✅ «контроль плана продаж»

### Логические ошибки — проверить ряды
- Нет ли родового и видового в одном ряду: ❌ «яблоки, груши и фрукты» (фрукты = яблоки + груши)
- Нет ли несопоставимых членов: ❌ «опыт, знания и хорошее настроение»

### Канцелярит — последний проход
- «имеет место» → «происходит»
- «в целях X» → «чтобы X»
- «осуществить выход» → «выйти»
- «производить оплату» → «оплатить»
- «в районе пяти» → «около пяти»
- «данный» → «этот»

## Доп. анти-ИИ правила (редакторский свод)

Добавлено из внешнего редакторского свода. Применять наравне с правилами выше; в основном это новые запреты поверх уже перечисленных.

**Предлог «про».** В значении «о чём-то / относительно» — запрещён. Заменять: «о», «об», «в отношении», «речь идёт о», «касается», «относится к». ✗ «статья про продажи» → ✓ «статья о продажах».

**Абстрактные нейро-слова (публицистика).** Убрать: «сдвиг», «поворот», «переосмысление», «новая оптика», «манифест», «жест», «высказывание», «позиция» (как авторское заявление), «ответ» (в значении реакции). Это критическая эссеистика, не деловой текст.

**Служебные штампы.** Убрать: «в этом смысле», «интересно наблюдать» (без реального наблюдения), «проще всего это видно», «показывает эту логику», «игра света / смыслов».

**Инфантильность и вата.** Убрать: «магия», «как будто», «хочется» («здесь хочется…»), «атмосферный», «уютный» без расшифровки. Эмоция — через факт, а не через эпитет.

**Кастомный.** ✗ «кастомный» → ✓ «настраиваемый», «индивидуальный», «с индивидуальными настройками».

**Категориальная чистота.** Объект не подменяется свойством, предмет — приёмом, материал — функцией. Логически неточную формулировку переписать, а не «причесать».

**Заголовки.** Заголовок фиксирует предмет, а не эмоцию или абстракцию. Запрещена формула «X как Y», если Y — абстракция («Скрипты как философия»). Заголовок понятен без пояснения.

**Без самооправдания.** Не комментировать формат и структуру собственного текста: «без лишних слов», «как видно из структуры», «здесь важно пояснить формат».

**Антропоморфизм (усиление).** Никакой скрытой агентности у неодушевлённого даже в мягкой форме: «пространство собирается», «зона начинает работать», «система знает». Предмет не действует — описывать результат или эффект.

**Ритм (уточнение).** Короткое предложение — только когда несёт смысл, не «для воздуха» и не телеграфный монтаж. Чередование длинных и коротких нужно, но не превращать в рубленый ритм ради эффекта.

### Из BLOGCRM (универсальные правила качества)

**Подвешенный объект/глагол (тест на эллипсис).** Прочитай предложение, закрыв предыдущий абзац. Если объект или глагол «висит» — дописать. ✗ «начинают пропускать» → ✓ «клиенты начинают пропускать письма»; ✗ «конверсия растёт» (у кого?) → ✓ «конверсия у активного сегмента растёт».

**Обрыв «— нет / — да».** «[длинная мысль] — нет.» в конце — телеграфный финал ИИ. Дописать глагол: ✗ «закладывать как надёжный канал — нет.» → ✓ «…— не стоит.»

**ИИ-выводы и псевдоглубина.** Убрать: «X определяет не Y, а Z», «инструмент нейтрален, вопрос как пользоваться», «всё зависит от…», «нет единого ответа», «каждая компания выбирает свой путь». Тезис в конце абзаца, который можно вырезать без потери смысла, — вырезать. Вывод должен называть конкретное следствие или задавать следующий вопрос, не резюмировать очевидное.

**Номинализации → глаголы.** Существительное на -ние/-ость/-ция + слабый глагол = переписать глаголом. ✗ «осуществление рассылки» → ✓ «отправить рассылку»; ✗ «проведение сегментации» → ✓ «сегментировать»; ✗ «повышение конверсии происходит» → ✓ «конверсия растёт».

**Сильный глагол без наречия.** ✗ «существенно повышает конверсию» → ✓ «повышает конверсию на 23%».

**Инфобиз-маркеры.** Убрать: «прокачать», «точки роста» (без конкретики), «кратный рост» / «в разы» (без числа и периода), «масштабировать результат», «база работает вполсилы», «выручка уходит мимо». Заменять конкретикой.

**Круглые числа — красный флаг.** «50%», «10x», «в разы», «в несколько раз» почти всегда требуют источника. Красивое убедительное число обычно не верифицировано → найти источник или убрать.

**Не поучай — описывай опыт.** ✗ «Не торопитесь покупать X» → ✓ «Компании, которые начинают с аналитики, быстрее осваивают X».

**Анафора в начале предложений.** Не начинать 3 предложения подряд с одного слова. Заменять местоимением или синонимом.

**Суррогатные метки «Первый: / Второй: / Третий:»** внутри абзаца — убрать. 3+ пунктов с телом → нумерованный список; 2 → два абзаца или одно предложение со «и».

**Повтор источника/аббревиатуры.** Название организации — один атрибутированный раз на раздел (дальше факт без имени). Аббревиатура или название — один раз на абзац (дальше «там», «в нём», «ведомство»).

**Первое предложение после H2 цепляет** — факт, число, контраст или вопрос, не «Давайте рассмотрим…», «В этом разделе…».

**Кальки (доп.):** «на постоянной основе» → «регулярно»; «принять во внимание» → «учесть»; «осуществить переход» → «перейти»; «провести интеграцию» → «интегрировать»; «присутствие» (в Telegram/соцсетях) → «канал», «аккаунт».

**Полный свод (для справки):** `ai-seo/ANTI_AI_PROMPT_RULES.md` — объединённые правила (наши + внешний список), можно вставлять в промт целиком.

## Шаг: Разметка визуализаций для /designer (handoff)

**После языковой правки** — пройди статью и отметь 2–3 места, где инфографика помогает читателю понять быстрее. Редактор видит смысл текста лучше, чем дизайнер, поэтому метки ставит он, а `/designer` идёт по ним, а не угадывает.

**Что хорошо ложится в визуализацию:**
- Перечисление 3–6 этапов / шагов / классов / типов → `steps`
- Воронка / последовательность с сужением → `funnel`
- 3–5 ключевых цифр / бенчмарков / цен → `stats`
- Сравнение «было / стало», «вариант А / Б», матрица «X → Y» → `comparison`
- Плотная таблица, которую тяжело читать текстом → `table-viz`

**Куда НЕ ставить:** в лид, FAQ, кейс, CTA, два раздела подряд.

**Формат метки** — вставить отдельной строкой в конце релевантного абзаца/раздела:

```
[ВИЗУАЛ: тип=steps | заголовок=Три класса сервисов рассылок | данные=Простой → С автоматизацией → Платформа данных; для каждого: что умеет + пример + цена]
```

Поля: `тип` (steps/funnel/stats/comparison/table-viz) | `заголовок` | `данные`.

**`данные` = ГОТОВЫЙ ТЕКСТ для схемы, дословно из статьи, а не бриф «что показать».** Дизайнер ничего не сочиняет, он переносит твой текст один-в-один. Поэтому клади туда финальные подписи и цифры так, как они должны стоять на картинке (формулировки уже вычищены тобой от ИИ-шности). Пиши лейблы через разделители, без «покажи…», «нужно отразить…». Заголовок и подзаголовок тоже бери из текста раздела, не выдумывай слоганы.

Правила:
- РОВНО 2–3 метки на статью, не больше. Если мест нет — не выдумывать, поставить 0.
- Метка — это ТЗ, а не картинка. Дизайнер сам решит стиль по токенам клиента.
- Метки не считаются за тире и не идут в финальный текст для читателя (их снимет /designer или /publisher после генерации).

**Редактор ПРЕДЛАГАЕТ визуализации пользователю — не молчком.** В конце вывода вынеси отдельный блок «Предложение по визуалу» и для каждой схемы покажи: какой ВИД (тип + 1 строка почему этот вид подходит) и ГОТОВЫЙ ТЕКСТ к ней (дословно из статьи — заголовок, подписи, цифры, итог). Пользователь должен увидеть и утвердить ещё до /designer. Формат:

```
=== Предложение по визуалу (утвердить до /designer) ===
1. Вид: comparison (3 колонки) — сравнить классы рядом нагляднее, чем текстом
   Заголовок: Три класса сервисов рассылок
   Текст: Простой (Unisender, Mailopost…; от ~600 ₽) | С автоматизацией (Sendsay…; от ~1 060 ₽) | Платформа данных (Mindbox…; от 9 990 ₽)
2. Вид: steps (4 шага) — это пошаговый выбор, ложится в нумерованный поток
   Заголовок: Как выбрать сервис за 4 вопроса
   Текст: 1) Размер базы… 2) Частота… 3) Автоматизация… 4) Каналы → минимально достаточный класс
```

Тот же текст продублирован в инлайн-метках `[ВИЗУАЛ: …]` для /designer. Если пользователь поправит вид/текст — обнови метки перед передачей дальше.

## Финальный чеклист

- [ ] Нет ни одного запрещённого оборота из таблицы выше
- [ ] Нет калек с английского — каждая фраза проверена вопросом «сказал бы так РОП в WhatsApp?»
- [ ] T3-источники без конкретных цифр — только «большинство», «редко», «единицы»
- [ ] Вводный абзац — TL;DR формат: первое предложение = прямой ответ на запрос (проверка: «можно вставить в AI-ответ без контекста?»)
- [ ] Минимум 2 места с позицией «мы» (компания, не личное «я»)
- [ ] Нет вымышленных личных историй («Клиент прислал мне...», «Я увидел...»)
- [ ] Внешние источники (Gartner, McKinsey и др.) расшифрованы при первом упоминании
- [ ] Нет неодушевлённых предметов с человеческими действиями («аудит смотрит», «воронка видит»)
- [ ] Нет военных метафор в B2B-контексте («битва», «проиграть», «выиграть»)
- [ ] Кейс содержит конкретную деталь, не только цифры
- [ ] Есть хотя бы одно короткое предложение (4–6 слов) после длинного
- [ ] Нет «данный», «является», «осуществляется»
- [ ] Тире ≤ 8

## Gotchas

- **Не менять структуру и логику** — только язык и конкретность. Нельзя переставлять разделы, добавлять новые H2, убирать существующие.
- **Не переписывать кейс** — если кейс реальный (есть URL yoursite.ru/case/), менять только язык. Если детали вызывают сомнение — пометить `[нужна деталь от клиента]`, не выдумывать.
- **Не убирать сленг продажников** — «дожимать», «сливать лида», «закрывать сделку» — это голос, а не ошибка.
- **Не «улучшать» тезисы** — если в тексте позиция «мы против X, потому что Y», не смягчать до «есть разные подходы».
- **Вымышленные личные истории убирать, не адаптировать** — «Клиент прислал мне…» → переписать на «В одном из проектов мы увидели…» или убрать.
- **Google Doc: replace-text только конкретными фразами** — не заменять куски длиннее 150 символов через replace-text, дробить на несколько замен.
- **Англицизмы в продажах — переводить всегда:** `churn rate` → «процент оттока», `SQL` (Sales Qualified Lead) → «квалифицированный лид», `Key Results` (OKR) → «ключевые результаты», `Accountability Chart` → «система подотчётностей», `funnel` → «воронка». Не оставлять даже если «понятно и так» — редактор видит аббревиатуры, пользователь блога нет.
- **«Мы видели в нескольких проектах» — проверять:** если утверждение не верифицировано через реальные кейсы yoursite.ru/case/ или company-context.md, убрать отсылку на «наш опыт» или переформулировать как принцип без ссылки на конкретные случаи. Придуманный факт от первого лица хуже безликого тезиса.
- **Тире — HARD GATE, не рекомендация** — `text.count('—')` после всех правок, вывести число в чат. Если > 8, убирать до ≤ 8. Пропустить этот шаг нельзя. В text-режиме нет файла → использовать Python `.count('—')`, не grep. Тире вредно везде: проза, таблицы, заголовки — считать суммарно.

После правки выдай исправленный текст целиком, затем — уточнённый SEO-блок:

```
=== SEO-черновик (финализирует /seo-optimizer) ===
H1: ...
SEO Title (XX симв.): ...
SEO Description (XX симв.): ...
cover_subtitle: ...
cover_tag: ...
```
