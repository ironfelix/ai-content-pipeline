# SEO Tech Audit

Ты проводишь техническую SEO-диагностику сайта. Пишешь результат в `<project>/research/tech-seo-audit.md` по формату эталона.

## Эталон формата

**ОБЯЗАТЕЛЬНО прочитай** перед началом работы:
`/Users/ivanilin/Documents/ivanilin/ai-seo/factor/research/tech-seo-audit-log.md`

Твой output должен повторять структуру: нумерованные разделы (robots, sitemap, индексация, мета, скорость), таблицы "было / делаем", явные рекомендации с обоснованием.

## Input contract

- `<project>/data/brief.md` существует и содержит URL, регион
- Доступ в интернет (WebFetch работает)

## Что проверяем (чеклист)

### 1. robots.txt
- Получи `https://<domain>/robots.txt` через WebFetch
- Проверь: дубли правил, некорректный синтаксис (`*?` без `/`), конфликты с sitemap, блокировка важных разделов (кейсы, услуги, блог)
- Проверь наличие `Sitemap:` директивы

### 2. sitemap.xml
- Получи sitemap (обычно `/sitemap.xml` или `/wp-sitemap.xml`)
- Проверь: наличие всех типов страниц, даты `lastmod`, отсутствие 404/redirect внутри sitemap
- Если CMS = WordPress — проверь `/wp-sitemap.xml` и поддеревья `/wp-sitemap-posts-*.xml`

### 3. Индексация
- Проверь `site:<domain>` через WebSearch (Яндекс и Google)
- Сравни количество страниц в индексе с количеством в sitemap
- Найди очевидные "мусорные" страницы (тестовые, черновики, технические)

### 4. Главная страница — тех. осмотр
Запусти `scripts/crawl.py <url>` чтобы получить:
- HTTP-статус, редиректы
- title, meta description, h1
- canonical
- наличие hreflang (если нужен)
- Open Graph
- количество h1 (должен быть 1)
- размер DOM, объём HTML

### 5. Ключевые разделы
Пройди те же проверки (п.4) для 3-5 ключевых страниц: блог, услуги, главная, контакты, типичная карточка продукта.

### 6. Core Web Vitals
Если доступен Google API key в env (`GOOGLE_PSI_API_KEY`) — запусти `scripts/pagespeed.sh <url>` для главной и ключевой страницы. Запиши: LCP, INP, CLS, TBT, общий скор.

Если ключа нет — пропусти, запиши "требуется GOOGLE_PSI_API_KEY для проверки CWV" и предложи пользователю добавить.

### 7. HTTPS и редиректы
- http → https
- www ↔ non-www
- трейлинг-слэш (`/page` vs `/page/`)
- Нет ли цепочек редиректов

### 8. Структура URL
- Кириллица/латиница
- Читабельность
- Вложенность
- Наличие параметров `?utm`, `?sid` в выдаче

### 9. GEO-готовность (LLM-видимость)

Базовые условия для индексации Яндекс Нейро, ChatGPT, Perplexity, Google AI Overviews.

#### 9.1. /llms.txt
- Получи `https://<domain>/llms.txt` через WebFetch
- Если 404 — записать как отсутствует. Рекомендуй создать: краткое описание сайта, ключевые разделы, для кого он. Формат свободный (как robots.txt, но для LLM).

#### 9.2. AI-боты в robots.txt
- В уже полученном robots.txt проверь наличие User-agent для: `GPTBot`, `ClaudeBot`, `PerplexityBot`, `GoogleBot-Extended`
- Если robots.txt содержит `User-agent: * / Disallow: /` — убедиться, что AI-боты не заблокированы
- Рекомендуй явно добавить разрешение: `User-agent: GPTBot / Allow: /` (и аналогично для ClaudeBot, PerplexityBot)

#### 9.3. Schema.org — типы разметки
- Проверь через crawl.py (поле `json_ld`) какие типы Schema уже есть
- Для контентного сайта (блог, статьи) нужны минимум:
  - `Article` / `BlogPosting` с полем `author` + `sameAs` к соцсетям автора
  - `Organization` на главной с `sameAs` к соцсетям компании
  - `FAQPage` на страницах с вопросами и ответами
  - `Person` с `jobTitle`, `worksFor` для страниц авторов
- Для товарного / коммерческого сайта:
  - `Product` + `AggregateRating` на карточках товаров
  - `LocalBusiness` или `Organization` на главной
  - `BreadcrumbList` для навигации
- Запиши: что есть / чего нет

#### 9.4. SSR/SSG — контент в HTML без JS
- Если CMS = React/Next.js SPA / Vue SPA — проверить через crawl.py: есть ли основной текст страницы в HTML или он появляется только после JS-рендера
- LLM-краулеры (GPTBot, ClaudeBot) не выполняют JavaScript → контент за JS = невидим для LLM
- Для WordPress / Tilda / статических сайтов — обычно ОК, отметить как "не применимо"
- Для Next.js — убедиться что включён SSR или SSG (не CSR)

#### 9.5. EEAT-блоки на статьях
- Проверь 2-3 ключевые статьи: есть ли фото автора, имя автора, дата публикации, ссылки на источники
- Без имени автора и даты — LLM не присваивает доверие к конкретному источнику

#### 9.6. TL;DR — прямой ответ в начале
- Проверь 2-3 ключевые страницы: начинается ли первый абзац с прямого ответа на запрос?
- Пример: запрос «что такое X» → первое предложение должно быть определением, а не вводным «в этой статье мы разберём...»
- LLM цитирует первые 1-3 абзаца — если они дают ответ сразу, шанс цитирования выше

## Интерактивные остановки

1. После сбора robots + sitemap + индексации — покажи 3-5 критичных проблем пользователю, спроси **"применяем фиксы сейчас или только документируем?"**
2. Если нашёл проблему, которую можно сразу починить (например, robots.txt), и есть SSH-доступ в brief — НЕ чини без явного подтверждения.

## Gotchas

- **Не выводить геолокацию из домена** — `.yaroslavl.ru`, `.msk.ru` и т.п. не говорят о регионе работы. Брать только из brief.md.
- **Не чинить без подтверждения** — нашёл проблему (robots.txt, redirect), есть SSH — никогда не вносить изменения без явного «применяем» от пользователя.
- **CWV без API ≠ пропуск раздела** — если `GOOGLE_PSI_API_KEY` не задан, написать явно: «CWV не проверены, требуется GOOGLE_PSI_API_KEY» и предложить добавить в .env.
- **Sitemap не всегда на `/sitemap.xml`** — для WordPress проверять `/wp-sitemap.xml`, для других CMS искать в robots.txt директиву `Sitemap:`.
- **`site:domain` через WebSearch — оценка, не точное число** — формулировать: «по оценке поисковика — около N страниц», не называть как факт.
- **Не добавлять рекомендации без наблюдения** — каждая проблема должна опираться на то, что реально увидел в HTML/robots/sitemap, не на общие предположения.
- **GEO ≠ переписать весь сайт** — в разделе 9 проверяем только технические условия доступности: боты, llms.txt, Schema-типы, SSR. Не нужно давать рекомендации по переработке контента в рамках тех. аудита.
- **/llms.txt 404 — не критично, записать как "рекомендуется"** — файл помогает LLM понять контекст сайта, но его отсутствие не блокирует индексацию. Это рекомендация, а не критическая ошибка.
- **AI-боты blocked — критично** — если `User-agent: GPTBot / Disallow: /` явно прописано в robots.txt, это блокирует индексацию ChatGPT и OpenAI-продуктов. Фиксировать как 🔴.
