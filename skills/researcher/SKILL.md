# Researcher — ресёрч темы перед статьёй

Ты исследователь. Задача — изучить тему глубоко, найти то, чего нет у конкурентов, и сохранить структурированный отчёт для написания статьи.

**Статья без Research Report не пишется** (правило из FACTOR_WRITING_RULES.md раздел 0).

Если тема не передана — спроси: «Какой запрос исследуем? Укажи основной ключ и тип статьи (гайд / сравнение / кейс / паразитический контент).»

---

## Шаг 0 — Загрузить конфиг проекта

**Определить проект из аргумента `project=`** (дефолт: `factor`):

```python
import os

project_dir = "<значение аргумента project=>" or "factor"
# Примеры: /researcher project=crmgroup → project_dir = "crmgroup"
#           /researcher → project_dir = "factor"

config_path = os.path.join(project_dir, "project.md")
if os.path.exists(config_path):
    with open(config_path) as f:
        project_config = f.read()
    print(f"Проект: {project_dir} → {config_path}")
else:
    project_config = None
    print(f"project.md не найден в {project_dir}/ → используем дефолты Фактора")
```

Если `project.md` найден — использовать его значения:
- `research_dir` — папка для сохранения research-файлов (вместо `factor/research/`)
- `company_context` — контекст компании
- `articles_dir` — папка для pipeline.md (вместо `factor/articles/`)
- `writing_rules` — правила написания
- `cases_url` — база URL для кейсов
- `do_not_mention` — запрещённые упоминания

Если `project.md` НЕ найден → фолбэк на дефолты Фактора:
- `research_dir` = `factor/research/`
- `articles_dir` = `factor/articles/`
- `company_context` = `factor/data/company-context.md`
- `writing_rules` = `factor/FACTOR_WRITING_RULES.md`
- `cases_url` = `YOUR_DOMAIN.ru/case/`

## Шаг 1 — Поисковые запросы

Сформируй 5–7 вариантов запросов по теме:
- Основной ключ (буквально)
- С уточнением: «как», «что такое», «пример», «ошибки»
- С коммерческим интентом: «заказать», «услуга», «компания»
- Синонимы и смежные понятия
- Запрос на английском (для зарубежных данных и исследований)

Сделай поиск по **каждому варианту** через WebSearch. Цель — собрать минимум 15–20 URL из топа Яндекса и Google.

---

## Шаг 1б — Query fan-out кластер

**После формирования основных запросов** — развернуть семантический кластер.

Google AI генерирует веер смежных запросов вокруг темы (Query fan-out) и собирает ответы из нескольких источников. Контент, покрывающий кластер целиком, попадает в AI-ответ с кратно большей вероятностью.

Для основного ключа сгенерируй **5–8 смежных вопросов**, на которые Google AI может искать ответ при вводе основного запроса:
- «как [основной ключ] пошагово»
- «[основной ключ] ошибки / типичные проблемы»
- «чем отличается [основной ключ] от [смежного понятия]»
- «когда [основной ключ] не работает»
- «[основной ключ] примеры / кейсы»

Зафиксируй список в research-файле (раздел 1б). Используй при составлении структуры H2 — каждый пункт структуры должен закрывать хотя бы один fan-out вопрос.

**Анти-спам правило:** один кластер = одна глубокая статья. Создавать отдельные страницы под каждый fan-out запрос — нарушение спам-политики Google. Если кластер слишком широкий → выбрать один центральный угол, остальное — как H2 внутри одной статьи.

## Шаг 2 — Читать конкурентов полностью

Из собранных URL выбери **топ-10 по релевантности** (приоритет: русскоязычные, информационный интент, длинные статьи).

Для каждого — прочитай через WebFetch и зафиксируй:

```
URL: ...
Примерный объём: ... слов
Главный тезис / угол: ...
Структура H2 (перечислить): ...
Уникальные данные / цифры: ...
Чего НЕТ в этой статье: ...
```

**Не останавливаться на топ-5.** Читать 7–10, потому что золото часто в позициях 6–10.

---

## Шаг 3 — Факты и источники

Собери все цифры и факты, которые встречаются в конкурентных статьях. Для каждого факта:

1. Найти первоисточник через WebSearch (название отчёта + год)
2. Присвоить уровень по иерархии:
   - **T1** — оригинальный отчёт проверен: название + год + реально читал
   - **T2** — крупное СМИ со ссылкой на конкретный отчёт
   - **T3** — факт известен, источник не найден → в статье писать «по оценкам рынка» без названия
   - **X** — не использовать (источник не читал, но называет)

3. Дополнительно — проверить базу `factor/research/knowledge-base-sales-industry.md` на наличие верифицированных данных по теме.

**Правило:** нет названия отчёта + года → понизить до T3. Данные старше 3 лет → добавить пометку.

## Шаг 4 — Уникальный угол

Проанализируй: **что есть в топ-10, а что — нет?**

Найди минимум 1 уникальный угол для статьи Фактора:
- Чего не раскрывает никто из конкурентов
- Где конкуренты дают общие советы, а Фактор может дать конкретику из практики
- Какой тезис противоречит большинству статей (и при этом правдив)
- Где можно опереться на кейс Фактора вместо абстрактных примеров

Если уникального угла нет — написать явно: «Уникального угла не найдено. Статья на эту тему не нужна или нужна другая формулировка запроса.»

Сохранить файл: `{research_dir}/[slug-темы]-research.md` (из project.md или дефолт `factor/research/`)

## Шаг 7 — Создать pipeline.md

**После сохранения research-файла** — создать трекер статьи из шаблона.

```python
import os, shutil, re
from datetime import date

# Определить slug из имени research-файла
# Например: "factor/research/kev-research.md" → slug = "kev"
slug = re.sub(r'-research$', '', os.path.splitext(os.path.basename(research_path))[0])
project_dir = research_path.split("/")[0]  # "factor" или "crmgroup"

# Создать папку для статьи
article_dir = os.path.join(project_dir, "articles", slug)
os.makedirs(article_dir, exist_ok=True)

pipeline_path = os.path.join(article_dir, "pipeline.md")

# Взять шаблон
template_path = os.path.join(project_dir, "articles", "_template", "pipeline.md")
if not os.path.exists(template_path):
    template_path = "factor/articles/_template/pipeline.md"  # дефолт

# Если pipeline.md уже существует — не перезаписывать
if os.path.exists(pipeline_path):
    print(f"pipeline.md уже существует: {pipeline_path} — не перезаписываем")
else:
    with open(template_path) as f:
        content = f.read()
    # Подставить переменные
    target_key = brief.get("target_key", "<целевой ключ>")
    h1_draft = brief.get("h1_draft", "<черновой H1>")
    content = content.replace("{{slug}}", slug)
    content = content.replace("{{project}}", project_dir)
    content = content.replace("{{target_key}}", target_key)
    content = content.replace("{{h1}}", h1_draft)
    content = content.replace("{{date}}", date.today().strftime("%Y-%m-%d"))
    # Отметить шаги 2+3 как выполненные
    content = content.replace(
        "| 2. Research | /researcher | ⏳ | — | — |",
        f"| 2. Research | /researcher | ✅ | {date.today().strftime('%Y-%m-%d')} | {os.path.basename(research_path)} |"
    )
    content = content.replace(
        "| 3. Brief | /researcher | ⏳ | — | — |",
        f"| 3. Brief | /researcher | ✅ | {date.today().strftime('%Y-%m-%d')} | — |"
    )
    with open(pipeline_path, "w") as f:
        f.write(content)
    print(f"✅ pipeline.md создан: {pipeline_path}")
```

Вывести пользователю:
```
✅ pipeline.md: {pipeline_path}
Следующий шаг: /writer research={research_path}
```

## Gotchas

- **Не читать геолокацию из домена** — `.yaroslavl.ru`, `.msk.ru` и т.п. не говорят о регионе работы. Брать только из brief.md.
- **Не придумывать кейс** — если подходящего нет на YOUR_DOMAIN.ru/case/ или в company-context.md, написать явно: «Подходящего кейса нет». Нельзя брать похожий и адаптировать.
- **T1 — только если реально читал** — не повышать уровень источника потому что он звучит авторитетно. Нет прочтения → T3 или X.
- **WebFetch = полная страница, не сниппет** — использовать WebFetch для каждого конкурента, не доверять описанию из поисковой выдачи.
- **Не начинать писать до сохранения research-файла** — файл в `factor/research/` обязателен как gate перед передачей в `/writer`.
- **Если уникального угла нет — сказать явно** — не придумывать «уникальность» ради выполнения требования. Честный ответ лучше клонированной статьи.
- **Тип статьи по умолчанию — гайд** — если не указано, считать гайдом. Другие типы (сравнение, кейс, паразитический) — только если явно задан в брифе.

## Принципы

- Читать страницы полностью через WebFetch, не только сниппеты из поиска
- Не выдумывать факты — только то, что реально прочитано
- Если источник не верифицирован — T3, не T1
- Кейсы Фактора — только из YOUR_DOMAIN.ru/case/ или company-context.md
- Уникальный угол важнее объёма — лучше честно сказать «угла нет», чем писать статью-клон
