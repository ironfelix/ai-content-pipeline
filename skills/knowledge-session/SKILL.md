# Knowledge Session — интервью с клиентом

Ты ведёшь онбординг-интервью нового клиента контент-пайплайна. Цель — за один сеанс собрать всё, без чего /writer выдаёт generic-текст: факты компании, границы продукта, язык клиентов, голос, кейсы.

**Это главный bottleneck онбординга** — без company-context.md пайплайн пишет «статьи ни о ком».

## Аргументы

```
/knowledge-session project=<slug>          → интервью для проекта <slug>
/knowledge-session project=<slug> doc=<id> → ответы уже есть в Google Doc/файле — распарсить вместо интервью
```

## Режимы

1. **Live-интервью** (дефолт): задавать вопросы блоками по 3–4, пользователь отвечает сам или транслирует ответы клиента с созвона. После каждого блока — фиксировать ответы, уточнять провалы.
2. **Парсинг готовых ответов**: если передан `doc=` или файл с ответами/транскриптом созвона — извлечь ответы на те же 20 вопросов, пометить пропуски `[нет ответа — спросить]`.

## Шаг 0 — Подготовка

1. Проверить, что проект существует: `{slug}/project.md`. Нет → предложить `./new-client.sh`
2. Прочитать шаблоны: `{slug}/data/company-context.md`, `{slug}/data/voice.md` (созданы скаффолдом)
3. **Перед интервью изучить сайт клиента** (WebFetch главная + услуги + кейсы + 2-3 статьи блога, если есть): половину фактуры можно снять с сайта, интервью тратить на то, чего на сайте НЕТ. Вопросы, ответы на которые уже найдены, задавать в формате подтверждения: «На сайте написано X — это так?»

## Шаг 1 — Интервью (5 блоков, ~20 вопросов)

### Блок A — Компания и продукт (≈10 мин)
1. Что вы делаете — объясните как другу, не как на сайте?
2. Сколько лет, сколько клиентов/проектов? Какими цифрами можно подтвердить опыт публично?
3. Из чего состоит продукт/услуга по этапам? Сроки, чек?
4. **Что вы НЕ делаете, хотя клиенты часто думают, что делаете?** ← граница продукта; самый недооценённый вопрос — без него статьи обещают то, чего компания не делает
5. Какие элементы продукта чаще всего пропускают/недооценивают?

### Блок B — Идеальный клиент (≈10 мин)
6. Опишите 2-3 реальных клиентов: кто по роли, размер бизнеса, ниша?
7. С какой формулировкой они приходят? **Дословно** — как говорят на первом созвоне?
8. Что у них «болит» на самом деле (≠ что они говорят)?
9. Что триггерит обращение именно сейчас (событие, сезон, кризис)?
10. Кому вы отказываете / кто плохой клиент?

### Блок C — Конкуренты и позиции (≈10 мин)
11. Кто конкуренты? Кого можно упоминать в статьях, кого нельзя?
12. Чем вы реально отличаетесь? (не «качество и подход» — что докажете?)
13. **С какими общепринятыми советами в вашей нише вы не согласны?** ← топливо для резких позиций в статьях
14. Что конкуренты делают хорошо? (честно)

### Блок D — Язык и голос (≈15 мин)
15. 10-15 слов профессионального сленга вашей ниши — как говорят свои?
16. Какие формулировки вас бесят в текстах про вашу сферу?
17. Кто будет автором статей по имени? Его роль, опыт, регалии?
18. 3-5 любимых фраз/оборотов автора — из реальной речи (попросить примеры с созвонов, из переписок)
19. Покажите 2-3 текста, которые «звучат как вы» (свои или чужие) — и 2-3 «точно не как вы»

### Блок E — Кейсы и факты (≈15 мин)
20. 5-10 лучших кейсов: ниша, что было, что стало, цифры. **Какие из них публичные (есть на сайте), какие нет?**
21. По каждому непубличному: можно ли использовать в статьях? С названием или анонимно?
22. Какие цифры/бенчмарки ниши вы знаете из практики (конверсии, циклы, чеки)? Что из этого можно публиковать?

## Шаг 2 — Записать результаты

Заполнить по шаблонам (НЕ перезаписывать структуру — заполнять секции):
- `{slug}/data/company-context.md` — блоки A, B, C, E (разделы 1-9)
- `{slug}/data/voice.md` — блок D (персона + грамматика + запреты + сленг + эталоны)
- `{slug}/project.md` — обновить `author_name`, `brand_voice` (1-2 строки), `do_not_mention`

Правила записи:
- Формулировки клиентов — **дословно, в кавычках** (это золото для лидов статей)
- Кейсы: таблица с колонкой «публичный/непубличный»; непубличные пометить «только с подтверждения»
- Незакрытые вопросы — списком в конце company-context.md: `## Открытые вопросы` с владельцем ответа
- В company-context.md проставить дату интервью

## Шаг 3 — Контроль полноты

Перед завершением проверить чек-лист (если пункт пуст — задать вопрос повторно, не закрывать сессию молча):
- [ ] Граница продукта (что НЕ делаем) — заполнена
- [ ] Минимум 3 дословные формулировки клиентов
- [ ] Минимум 2 резкие позиции (с чем не согласны)
- [ ] Минимум 10 слов сленга
- [ ] Минимум 3 кейса с цифрами и пометкой публичности
- [ ] Автор по имени + 3 его фразы
- [ ] do_not_mention заполнен

Вывести пользователю: что собрано, что осталось открытым, следующий шаг (`/marketing-audit` или `/researcher project={slug}`).

## Gotchas

- **Не превращать в анкету.** Вопросы — скелет; если клиент в ответе на №7 рассказал про конкурентов — зафиксировать в блок C и не спрашивать повторно.
- **«Как на сайте» — переспрашивать.** Если ответ совпадает с маркетинговым текстом сайта («индивидуальный подход») — попросить пример из последнего проекта.
- **Цифры без подтверждения не записывать как факты.** Помечать `[со слов клиента, не публиковать без проверки]`.
- **Голос ≠ список запретов.** Главное из блока D — персона с характером (см. voice.md раздел 1). Запреты вторичны.
- **60 минут — это лимит клиента, не твой.** Если интервью идёт письменно/асинхронно — можно глубже.
